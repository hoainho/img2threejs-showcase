# Contributing a showcase demo

This is the flow to get a model built with [img2threejs](https://github.com/hoainho/img2threejs)
onto the live gallery. It ends in a normal GitHub PR against `main` — this doc, the
[Issue Form](https://github.com/hoainho/img2threejs-showcase/issues/new/choose), and the PR
template all ask the same questions so nothing gets lost between steps.

## Steps

1. **Prerequisites**: Node 20, a GitHub account, and a `createXModel.ts` already generated by the
   img2threejs skill (see its [SKILL.md `Output` section](https://github.com/hoainho/img2threejs#output))
   — reconstruction must be code-only, not a downloaded mesh or art pack.
2. **Fork + clone** this repo, then `npm install`.
3. **Add your demo** as a *new* entry (do not edit other demos' entries):
   - drop the factory file under `src/demos/<id>/` (keep the exported function name from the
     skill's output, e.g. `createXModel`);
   - add a new object to the `demos` array in `src/demos/registry.ts` — every `DemoEntry` field is
     required (TypeScript will refuse to build otherwise), and set `status: 'final'`. Fill
     `author` (your display name) and `authorUrl` (usually your GitHub profile) — this is what
     shows as the "by <author>" byline on the gallery card and demo page, so contributors are
     credited by name;
   - add the reference image to `public/references/<id>.<ext>`.
4. **Verify locally**: `npm run build && npm run preview`, open `#/demo/<id>`, and sanity-check the
   camera, lighting, and orbit controls.
5. **Self-check** against [What gets a submission approved](#what-gets-a-submission-approved) below.
6. **Optional but recommended** for anything non-trivial: open the
   [**Submit a showcase demo**](https://github.com/hoainho/img2threejs-showcase/issues/new/choose)
   issue form first to get early feedback on scope/fit before you spend time on the PR.
7. **Push your branch and open the PR** — either follow the compare link in the issue form /
   above, or run `gh pr create --fill`. The description auto-fills from the PR template; fill in
   the contributor section.
8. **CI runs automatically** (`pr-safety-check`: typecheck, build, and a static safety scan) and
   must be green before a maintainer reviews. On merge, `deploy.yml` republishes GitHub Pages
   automatically — there's no manual deploy step.

## What gets a submission approved

Each rule below is a pass/fail test a maintainer (and, where noted, CI) actually checks:

- **Content-origin**: the reference image is one you have rights to use (own photo, public
  domain, or explicitly licensed) — state the source in the PR. No photoreal likeness of a
  private individual without consent.
- **Code-only**: the factory builds geometry procedurally in TypeScript (primitives / `Shape`
  extrude / curve+tube / instancing / canvas textures) — no imported `.gltf`/`.obj`/`.fbx`, no
  downloaded texture packs. This is img2threejs's own "code-only, not photogrammetry" promise.
- **No-network** *(checked by CI)*: zero runtime network calls, no external CDN or fonts — the
  same constraint already stated for the rest of this site.
- **Naming / registry-completeness** *(checked by CI + the TypeScript build)*: `id` is kebab-case,
  unique, and matches its `src/demos/<id>/` folder exactly; every `DemoEntry` field is filled;
  `status` is `'final'`.
- **Asset size/type** *(checked by CI)*: reference image ≤ 800 KB, `.png`/`.jpg`/`.jpeg`/`.webp`
  only — no `.svg` for reference images (inline `<script>` in SVG is a real XSS vector, so it's
  rejected outright, not reviewed case-by-case).
- **Scope of change** *(enforced via `CODEOWNERS`)*: the PR only touches `src/demos/**`,
  `public/references/**`, and appends to `src/demos/registry.ts`. Anything outside that surface
  (build config, workflows, shared runtime code) requires the repo owner's review and is unlikely
  to be approved from a first-time contributor without prior discussion — open an issue first.

## CI: `pr-safety-check`

Runs on every PR against `main`: `npm run build` (typecheck + Vite build), then a static scan
(`scripts/check-showcase-safety.mjs`) for banned runtime patterns (`fetch`, `eval`, external URL
literals in demo code, etc.), reference-image size/type, and a registry-id ↔ folder cross-check.
It's a static scanner, not a sandbox — a determined bad actor could still get past it with
obfuscated code. The actual backstop is human review plus `CODEOWNERS` on anything sensitive.
