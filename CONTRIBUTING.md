# How to contribute a showcase demo

This turns a model you generated with [img2threejs](https://github.com/hoainho/img2threejs)
into a live entry on the [gallery](https://hoainho.github.io/img2threejs-showcase/).
Every step below has a copy-paste command — you shouldn't need to write anything
from scratch except the factory code itself and the demo's description.

## TL;DR

If you already have a generated model and just want the commands:

```bash
# 1. fork + clone (replace <you> with your GitHub username)
git clone https://github.com/<you>/img2threejs-showcase.git
cd img2threejs-showcase
npm install

# 2. branch
git checkout -b add-demo/<id>

# 3. scaffold — creates src/demos/<id>/ and a registry.ts entry, all TODO-marked
npm run new-demo -- <id> "<Title>" object   # or "character"

# 4. paste your img2threejs factory into the generated file, fill every TODO
#    in registry.ts, add public/references/<id>.png — see step 4 below

# 5. verify
npm run build && npm run preview
node scripts/check-showcase-safety.mjs --base main

# 6. ship it
git add -A
git commit -m "Add <id> showcase demo"
git push -u origin add-demo/<id>
gh pr create --fill
```

Read on for what each step actually does and why — especially step 4, since
that's the only part the scaffold can't do for you.

## Step by step

### 0. Before you start

You need:
- Node 20 and a GitHub account
- A model already generated by the [img2threejs](https://github.com/hoainho/img2threejs)
  skill — a `createXModel.ts` factory. Reconstruction must be **code-only**:
  no imported `.gltf`/`.obj`/`.fbx`, no downloaded texture pack.

### 1. Fork and clone

```bash
gh repo fork hoainho/img2threejs-showcase --clone
cd img2threejs-showcase
npm install
```

No `gh` CLI? Click **Fork** on the [repo page](https://github.com/hoainho/img2threejs-showcase),
then `git clone https://github.com/<you>/img2threejs-showcase.git`.

### 2. Branch off `main`

```bash
git checkout -b add-demo/<id>
```

Pick `<id>` now — it's kebab-case (lowercase, hyphens), unique, and becomes
the URL: `#/demo/<id>`. Example: `rubber-duck`, `viking-shield`.

### 3. Scaffold the demo

```bash
npm run new-demo -- <id> "<Title>" object   # subjectClass: object | character
```

This runs [`scripts/new-showcase.mjs`](scripts/new-showcase.mjs) and does two things:
- creates `src/demos/<id>/create<Id>Model.ts` — a placeholder gray box, so
  `npm run build` and `#/demo/<id>` already work before you touch anything
- inserts a new entry into the `demos` array in `src/demos/registry.ts`,
  with every field either filled in or marked `TODO`

Nothing else changes — no other file needs editing.

### 4. Fill in the two things the scaffold can't write for you

**a. The factory.** Open `src/demos/<id>/create<Id>Model.ts` and replace the
placeholder box body with your img2threejs output, keeping the exported
function name (or rename it in both places).

**b. The registry entry.** Open `src/demos/registry.ts`, find the block the
scaffold added (it's at the top of the `demos` array), and replace every
`TODO`:

| Field | What goes here |
|---|---|
| `blurb` | 1-2 sentences — what makes this reconstruction interesting, matching the tone of existing entries |
| `generatedWith` | e.g. `'img2threejs v1.2'` — the skill version you used |
| `author` | your display name — shows as "by \<author\>" on the card and demo page |
| `authorUrl` | usually your GitHub profile |
| `status` | change `'placeholder'` → `'final'` once the real factory is in |
| `cameraPosition` / `cameraTarget` / `cameraFov` | tune these in `npm run preview` until the framing looks right |

**c. The reference image.** Add `public/references/<id>.png` (or `.jpg`/`.jpeg`/`.webp`,
**≤ 800 KB** — larger files and other formats, including `.svg`, are rejected by CI).

### 5. Verify before you push

```bash
npm run build && npm run preview   # then open #/demo/<id> in the browser
node scripts/check-showcase-safety.mjs --base main
```

The second command is the exact static-safety scan CI runs on your PR — running
it locally first means no surprises. Take a screenshot of the rendered demo
while you're here; you'll attach it to the PR.

### 6. Push and open the PR

```bash
git add -A
git commit -m "Add <id> showcase demo"
git push -u origin add-demo/<id>
gh pr create --fill
```

No `gh` CLI? Push, then open the [compare page](https://github.com/hoainho/img2threejs-showcase/compare)
and pick your branch. Either way, the PR description auto-fills from
[`PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md) — fill in the
contributor section (screenshot, rights statement, author).

Optional: open the [**Submit a showcase demo**](https://github.com/hoainho/img2threejs-showcase/issues/new/choose)
issue form *before* doing the work, if you want early feedback on scope.

### 7. What happens next

`pr-safety-check` runs automatically (build + the same safety scan from step 5)
and must pass before a maintainer reviews. A PR that only touches
`src/demos/**`, `public/references/**`, and `src/demos/registry.ts` doesn't
need special sign-off beyond normal review; anything outside that requires
the owner's review, enforced by [`.github/CODEOWNERS`](.github/CODEOWNERS).
On merge, `deploy.yml` republishes the gallery automatically.

## What gets a submission approved

Each rule below is a pass/fail test — most are checked automatically, some
need a maintainer's judgment:

| Rule | Checked by |
|---|---|
| Reference image is one you have rights to use (own photo, public domain, or licensed) — name the source in the PR | maintainer |
| Factory builds geometry procedurally in code — no imported mesh, no downloaded texture pack | maintainer |
| Zero runtime network calls, no external CDN/fonts | CI (`check-showcase-safety.mjs`) |
| `id` kebab-case, unique, matches its `src/demos/<id>/` folder exactly | CI + `tsc` |
| Every `DemoEntry` field filled, `status: 'final'` | `tsc` (all fields required) |
| Reference image ≤ 800 KB, `.png`/`.jpg`/`.jpeg`/`.webp` only (no `.svg`) | CI |
| PR only touches the contribution surface above, or has owner review | `CODEOWNERS` |

## Troubleshooting

- **`npm run new-demo` fails with "already exists"** — pick a different `<id>`,
  or if you're re-running after fixing a mistake, delete `src/demos/<id>/`
  and remove its entry from `registry.ts` first.
- **`check-showcase-safety.mjs` flags a URL literal** — it only allows
  `http(s)://` strings inside `registry.ts`'s own metadata fields
  (`sourceUrl`, GitHub links). Any URL in your factory code itself, even in
  a comment, needs to go.
- **Build fails on `registry.ts`** — almost always a missing `DemoEntry`
  field; the error message names which one.

## CI: `pr-safety-check`

It's a static scanner, not a sandbox — it can't catch every possible
obfuscated bad pattern. The real backstop is human review plus
`CODEOWNERS` on anything sensitive, not the scanner alone.
